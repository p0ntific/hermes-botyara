# Hermes Botyara: Agent Context

## Scope

- This repository is the only active Hermes codebase. Do not restore scripts from its parent directory.
- Product: Pulsar (`https://pulsar-tg.ru/`), an AI service that scans selected Telegram chats, finds messages from potential customers, and forwards them to the client.
- Primary business goal: determine whether the lead is interested in receiving customers in this format, then hand warm leads to a human manager.
- Communicate with the owner in Russian, directly and concisely.
- Never print or commit Telegram sessions, API keys, bot tokens, proxy credentials, or production `.env` values.

## Sales Behavior

The dialog pipeline is:

`history -> classify stage -> decide action -> render reply -> persist/notify`

Stages:

- `primary_interest`
- `needs_explanation`
- `qualification_needed`
- `objection_without_commitment`
- `ready_to_test`
- `meeting_agreed`
- `contact_or_later`
- `not_interested`
- `negative_or_non_target`
- `unknown`

Actions:

- `send_reply`
- `handoff_to_manager`
- `silent_stop`
- `manual_review`

Required behavior:

- `ready_to_test`, `meeting_agreed`, and `contact_or_later` -> send the canonical handoff, notify the manager chat once, set `warm_notified`, and stop automatic replies.
- `not_interested` and `negative_or_non_target` -> do not reply, set `stopped`, and stop automatic replies.
- Low-confidence `unknown` -> no client reply; notify for manual review.
- `primary_interest` -> briefly explain Pulsar, provide the website, mention that the manager helps choose chats and criteria, then ask only whether this lead-search format is interesting.
- `objection_without_commitment` -> reduce risk with the free test and assisted setup; do not hand off without interest.
- If the client asks a concrete question while otherwise ready, answer the question first. Do not hand off in that same message.
- A short affirmative answer (`Да`, `Да, интересно`, `Давайте`, including a greeting) to the bot's question about interest in the format is deterministic `ready_to_test`, even if the LLM calls it `primary_interest`.
- If a generated reply promises that a manager will contact/write to the client, force `handoff_to_manager` and manager notification. Never promise contact without notifying.
- Do not ask which chats, customers, requests, niches, or criteria the lead wants. Maxim/Andrew handles that after handoff.
- Do not put the website in the first cold message. It is allowed after the lead shows initial interest.
- Do not write supportive filler such as "понимаю вас". Keep replies short, concrete, and sales-directed.

Canonical follow-up meaning:

> Pulsar scans selected Telegram chats, selects messages from potential customers, and sends them to the client. The manager helps select chats and criteria. Ask whether receiving customers in this format is interesting.

Canonical handoff is generated by `SalesBrain.handoff_message()` with the account-specific manager username. Keep it short and do not add qualification questions.

Regression examples:

- `интересно, расскажите` -> explain the format; no manager yet unless it directly answers a prior format-interest question.
- `Да` / `Да, интересно` after the format-interest question -> warm handoff and notification.
- `с радостью попробую, но боюсь` -> warm handoff.
- `боюсь, пока не готов` -> objection reply, no manager.
- `давайте созвонимся` -> warm handoff.
- `не интересно` -> silent stop.
- Aggression, bot reply, or self-promotion -> silent stop.

Historical failures fixed by regression guards:

- `aleksandr_tgrek` and `shashkov_smm`: short affirmative replies were incorrectly left at `primary_interest`.
- `maria_shutovskaya`: the bot promised manager contact without notifying; the client later complained that nobody contacted her.
- Old logs before the fixes can show these behaviors and must not be treated as current expected behavior.

## Account Routing

Production has two Telegram sender accounts sharing one durable queue:

| Config name | Sender identity | Manager for warm leads | Cold DM daily limit |
| --- | --- | --- | --- |
| `main` | Andrew's account | `@MaksIgitov` | 10 |
| `personal` | Maxim's account | `@andrew_pontific` | 10 |

`manager_username` means the human receiving the warm lead, not the Telegram account that sent the cold message.

The same account that sent the cold message must continue that lead's dialog. Workers must ignore leads owned by another account.

Both YAML accounts inherit the global `PROXY_URL` unless an account has its own `proxy:` value. Telegram StringSession endpoints are normalized to port `443`. These details are required in production: direct Telegram connections from the host have timed out.

## Code Map

- `tg_lead_skill.py`: production launcher; keep it.
- `hermes/app.py`: startup, worker supervision, dispatcher wiring.
- `hermes/worker.py`: one Telethon client per account, incoming replies, sends, and notifications.
- `hermes/dispatcher.py`: shared queue assignment, capacity, limits, and cooldowns.
- `hermes/sales.py`: stage guards, action decisions, rendering, handoff, and notification text.
- `hermes/prompts.py`: cold pitch, classifier, and reply prompts.
- `hermes/store.py`: SQLite state, transcripts, queue, account health, and LLM logs.
- `hermes/notify.py`: manager notification via Bot API with MTProto fallback.
- `hermes/llm.py`: OpenAI-compatible provider routing and fallback.
- `hermes/config.py`: environment and multi-account YAML loading.
- `hermes/transcript.py`: production inspection CLI.
- `test_sales_flow.py`: sales and private-reply regressions.
- `test_infra.py`: account, dispatcher, storage, and LLM routing tests.

Avoid adding LangChain or SalesGPT as a dependency. The project intentionally uses only the SalesGPT-style stage-machine principle in a small local module.

## Persistence And Statuses

- Production state is SQLite `~/.hermes/hermes.db` unless `HERMES_DB` overrides it.
- Legacy `mtproto_leads.json` is migration input only.
- Important lead fields: `status`, `last_stage`, `last_action`, `manager_notified_at`, `stop_reason`, `reply_count`, and owning `account`.
- Terminal statuses are `warm_notified`, `stopped`, and `manual_review`; automatic replies must not continue after them.
- Manager notification includes username, account, stage, action, confidence, reason, model, last client message, and conversation history.

Inspection commands from `~/.hermes`:

```bash
python3 -m hermes.transcript leads --limit 100
python3 -m hermes.transcript show <username>
python3 -m hermes.transcript queue
python3 -m hermes.transcript llm --limit 100
```
